# Shared rails — planning matrix (RedDirt)

**Packet FND-1.** Practical view of **rails** the unified campaign engine will standardize. Use with [`unified-campaign-engine-foundation.md`](./unified-campaign-engine-foundation.md).

| Rail | Purpose | Current repo evidence | Current gaps | Likely first implementation step | Dependent workbenches / domains | Risk if delayed |
|------|---------|------------------------|--------------|----------------------------------|---------------------------------|-----------------|
| **Incoming work (UWR-1+)** | One mental model and **handles** to “open work” for triage and CM | **UWR-1:** `open-work.ts` merged `UnifiedOpenWorkItem` (email, intake, task) + workbench read-only block; `Submission` / `ArkansasFestival*`, `InboundContentItem` still **outside** v1; `IncomingWorkMatrix` doc | **No** one table; **not** all matrix sources in one list | **UWR-2+:** more sources, filters (county), **“for me”** on hub when product wants it | CM shell, all “queue” UIs, intake | Solo operator **drowns**; CM cannot scale if unified read stalls after three models |
| **Identity / contact** | Every touch resolves to a **known person** | `User` (email unique, merge), `VolContact`, `VoterFileContact`, `linkedVoterRecordId` on `User` | **Not** every form creates `WorkflowIntake`; duplicate **intents** in different tables | **Idempotent** create/update from **all** public capture paths; document **one** join path: form → `User` → intake (optional) | Comms, volunteer, all forms | **Fragmented** CRM; **double** work |
| **Assignment / ownership (ASSIGN-1+)** | Who acts next; *future* which **seat** | `assignedToUserId` on `EmailWorkflowItem`; `assignedUserId` on `WorkflowIntake`, `CampaignTask`, `CommunicationThread` (see `User` relations in schema) | **No** `positionId` on rows; **not** a separate “role” column | UWR-1: `getOpenWorkForUser` / `getUnassignedOpenWork` (three sources); `getOpenWorkCountsBySource` (health); position inbox **narrative** | CM, all queues, delegation | **No delegation** if assignment stays user-only; roll-up **undefined** in code without seating |
| **Status / priority** | Done vs not; urgency | Prisma `enum`s per model; DTOs | **Inconsistent** labels (“pending” vs “in_review”) | **Glossary** in docs + DTO mappers; optional **unified** `OpenWorkStatus` for read model only | All | **Wrong** sort order; **false** “clear” |
| **Provenance / audit** | Why a row looks like it does; **automation/AI** trail | `metadataJson` (email workflow, festival suggest `decisionRationale`); `EmailWorkflowAuditLog` | **Not** all domains; **ad hoc** key names in JSON | **Namespace** rules (`_emailWorkflow*`, `automation*`) in one doc; log table **pattern** for new automation | email-workflow, future AI | **Trust** and **compliance** fail |
| **Automation & policy** | Safe machine actions | **Comms** send pipeline + **queue-first**; `extension-points.ts`; `email-workflow` review states; `process-env` in cron routes; webhooks in `comms/execute` and **inbound** | **No** one **policy** engine; comms and email **workflow** are **separate** products | **Do not** merge sends; add **if** a step runs **only** in review → mark in metadata; document **invariants** in foundation | All outbound, intake auto-routing | **Accidental** send or **silent** failure |
| **Recommendation / AI assist (digital CM brain core)** | Suggest, not commit; shared OpenAI stack: `client.ts` + `embeddings.ts` + `search.ts` + `prompts.ts` + `assistant/` (RAG, tools, public guide) | Also: `comms/ai` on `CommunicationThread` (`aiThreadSummary` / `aiNextBestAction`); E-2 heuristics + E-3 extension stubs on `EmailWorkflowItem`; `classifyIntake` on forms; `ConversationOpportunity` queue | Provenance not unified for every LLM call; `social-analytics-ai-stubs` unfilled | BRAIN-1+: register touchpoints; keep advisory; no auto-send, auto-role, or PII exfil | All domains + TALENT-* | Uncontrolled automation; duplicate prompt policies if new islands bypass `prompts.ts` |
| **Alignment + override learning (ALIGN-1+)** | Governed campaign alignment (mission, voice, red lines) as **layered** context, not one blob; human divergences from suggested paths are **observable** learning inputs | `prompts.ts` + `SearchChunk` + E-2 `metadataJson` as de facto sources; E-2B `overwrittenFields` / `preservedOperatorFields` pattern; types in `alignment.ts`, `overrides.ts`, `user-context.ts` | No `CampaignAlignmentVersion` table; no append-only `AutomationOverrideEvent` store; RAG not filtered by workbench | ALIGN-2: map layers → prompts/chunks; OVR-1: first override log on one touchpoint; keep causality honest | AI touchpoints, CM policy, TALENT training | If skipped: model “sounds on-brand” by accident; no audit trail when operators fix bad suggestions; future RBAC + personalization becomes risky without boundaries doc |
| **Geography / county** | **Local** field and comms | `County` model, `User.countyId`, `ArkansasFestival*`, workbench DTOs | Ingest sometimes **lacks** county; **fuzzy** in festival | Enforce or **suggest** county on **high**-impact creates; show **filter** in unified list | Workbench, field, festival | **Wrong** door-knock or **messaging** |
| **Content & comms** | “**What we said**” and **outbound** control | `CommunicationThread`, `CommunicationSend`, `CommunicationPlan`, `CommunicationRecipient*`, `extension-points`, workbench DTOs | **Tier-2** **split**; email **workflow** is **triage** not “send all” | **Link** from email workflow to **no** send; **link** to thread when a send exists; **one** DTO for “outbound state” in CM view | comms, email workflow, public | **Double** comms, **inconsistent** state |
| **Calendar / timeline** | **When** and **sequencing** | `CampaignEvent`, workbench, `EventRequest`, `Calendar` route, tasks with start/end, comms with window | **Three** “event” entry points (request, ingest, admin) | **Event** as **row**; **festival** and **local** as **sourced**; CM **upcoming** from **one** read | Field, workbench, comms | **Missed** windows; **clash** |
| **Talent / adaptive training (TALENT-1)** | **Advisory** development + capability signals on **operational** objects; **not** HR | `CampaignTask` outcomes, time-to-action on `assignedToUserId` rows, triage fields on `EmailWorkflowItem`, **(future)** training completions; **docs** [`talent-intelligence-foundation.md`](./talent-intelligence-foundation.md) | **No** central observation log; **no** LMS; **no** per-user talent UI | **Types** in `src/lib/campaign-engine/talent.ts` + `training.ts`; **TALENT-2:** append-only observation contract | All positions (ROLE-1); CM + managers | **Wrong** people get **trust** **expansion** without evidence; **or** no development path at scale |

### How to use this matrix in packets

- **For each** new feature, **name** the rail it touches; if **new** rail, **add** a row and **FND-1**-style **evidence**.
- **“First step”** should stay **small**: **read** path or **doc** or **one** new column—**not** a framework rewrite.
- **Campaign Manager (CM) packets** should **read** this matrix in **orchestration** and **triage** design.
