### Build protocol rails (matrix extension)

All feature packets should respect:

- **No build** without **lane status** (honest **L0–L5** in the ledger or packet notes).  
- **No large** feature **blind to** **division** **next** **stage** — **registry** **forward** **path** or same-packet **path** **contract** (**BLUEPRINT-EXP-1**).  
- **Unattended** **runs** follow [**AUTO-BUILD-1**](./AUTO_BUILD_PROTOCOL.md) / **AUTO-BUILD-2** (nightly **workflow**) — **not** a license for **sends**, **migrations** (unless **assigned**), or **auth** **changes** without **human** **review**; the **nightly** **Action** only **preflight**-**writes** + **typecheck** + **artifact**.  
- **No automation** without **approval policy** (explicit packet + product alignment).  
- **No AI authority tier** — advisory and provenance stay explicit.  
- **No hidden outbound actions** — queue and execution paths stay legible.  
- **No untracked data mutation** — provenance / audit patterns per domain.  
- **Every new lane** must define **source of truth** and **operator review path**.  

### Division balance rail (DIV-OPS-1)

- **No** system **expansion** (new automation, L4+ features, large UI) without **checking** **division** **parity** in [`DIVISION_MASTER_REGISTRY.md`](./DIVISION_MASTER_REGISTRY.md) + **Blueprint Progress Ledger** ([`PROJECT_MASTER_MAP.md`](./PROJECT_MASTER_MAP.md)).  
- **No** **advanced** **automation** in one division if **dependent** foundations sit **below L2** (unless the packet **documents** the exception and follow-up).  
- **Every** build **must** state **which division(s)** it **advances** and include **imbalance** notes in the **division status** return (see **PROTO-2** / **DIV-OPS-1**).

### Build steering rail (DIV-OPS-2)

- **Every** packet must **declare** a **target** division, **reason**, and **non-target** divisions (**Build steering decision** — see [`BUILD_PROTOCOL_AND_BLUEPRINT_AUDIT.md`](./BUILD_PROTOCOL_AND_BLUEPRINT_AUDIT.md)). **No** **silent** reprioritization.  
- **No** defaulting to **“easier”** builds (extra **Comms** / **Workbench** / **Data** work) when **CRITICAL** / **HIGH** **Priority** divisions in the **registry** are **behind** — **unless** the **return** **explicitly** justifies **skipping** them.  
- **System balance** **overrides** **developer convenience**; **ChatGPT** sets direction; **Cursor** may **propose** an alternative but **not** **assume** it.  

### Blueprint must lead (BLUEPRINT-EXP-1)

- **Future** **state** for a **division** should be **defined** in the **registry** **before** **large** **implementation** (or in the **same** packet with **explicit** **path** **acceptance**).  
- **No** **blind** **building** — **next** **stage** / **unlocks** / **dependencies** **named** in [`DIVISION_MASTER_REGISTRY.md`](./DIVISION_MASTER_REGISTRY.md) and **tied** to [`PROJECT_MASTER_MAP.md`](./PROJECT_MASTER_MAP.md) **Future-state blueprint** where **material**.  
- Pairs with **Build steering** and **Packet scaling**: **unlocks** may **legitimately** **steer** work **ahead** of **pure** **imbalance** when the **horizon** is **documented** (see **§0.10** in [`THREAD_HANDOFF_MASTER_MAP.md`](./THREAD_HANDOFF_MASTER_MAP.md)).  

### How to use this matrix in packets

- **For each** new feature, **name** the rail it touches; if **new** rail, **add** a row and **FND-1**-style **evidence**.
- **“First step”** should stay **small**: **read** path or **doc** or **one** new column—**not** a framework rewrite.
- **Campaign Manager (CM) packets** should **read** this matrix in **orchestration** and **triage** design.
